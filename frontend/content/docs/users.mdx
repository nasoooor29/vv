# User Management

Administrators can manage user accounts, credentials, and permissions through the user management interface.

## Overview

User management allows you to:
- Create and delete user accounts
- Update user information
- Assign and manage roles/permissions
- View all registered users

## Required Permissions

| Action | Permission |
|--------|------------|
| All user operations | `user_admin` |

## Accessing User Management

Navigate to **System** > **Users** in the sidebar.

## User Operations

### Viewing Users

The users table displays all registered users with:
- Username
- Email
- Role/permissions
- Created date
- Actions

### Creating a User

1. Click **Create User**
2. Fill in the user details:
   - **Username**: Unique username
   - **Email**: Unique email address
   - **Password**: Initial password
   - **Role**: Initial permissions (comma-separated)
3. Click **Create**

### Editing a User

1. Click the **Edit** button on a user row
2. Modify user details:
   - Username
   - Email
   - Password (optional, leave blank to keep current)
3. Click **Save**

### Managing Roles

1. Click the **Roles** button on a user row
2. Toggle permissions using checkboxes
3. Click **Save**

Available permissions are organized by category:
- **Docker**: `docker_read`, `docker_write`, `docker_update`, `docker_delete`
- **QEMU**: `qemu_read`, `qemu_write`, `qemu_update`, `qemu_delete`
- **System**: `event_viewer`, `event_manager`, `settings_manager`, `audit_log_viewer`, `health_checker`
- **Admin**: `user_admin`

### Deleting a User

1. Click the **Delete** button on a user row
2. Confirm the deletion

**Note**: Deleting a user also invalidates all their active sessions.

## API Reference

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/api/users` | GET | Get all users |
| `/api/users/:id` | GET | Get user by ID |
| `/api/users` | POST | Create new user |
| `/api/users/:id` | PUT | Update user |
| `/api/users/:id` | DELETE | Delete user |
| `/api/users/:id/role` | PATCH | Update user's role |

### Create User Request

```json
{
  "username": "newuser",
  "email": "newuser@example.com",
  "password": "securepassword123",
  "role": "docker_read,qemu_read"
}
```

### Update Role Request

```json
{
  "role": "docker_read,docker_write,docker_update,qemu_read"
}
```

## OAuth Users

Users created via OAuth have:
- Username derived from their email
- Random generated password (not used for login)
- Default minimal permissions

OAuth users can only log in via their OAuth provider.

## Password Security

- Passwords are hashed using bcrypt
- Minimum password requirements should be enforced
- Passwords are never returned in API responses

## Best Practices

1. **Principle of Least Privilege**: Grant only necessary permissions
2. **Regular Audits**: Review user permissions periodically
3. **Strong Passwords**: Enforce strong password policies
4. **Remove Inactive Users**: Delete accounts no longer in use
5. **Separate Admin Accounts**: Don't use admin accounts for daily operations
